## 1. Robot Setting（System Prompt）

> **正本：[`eap_import_bundle/robot_setting.txt`](../eap_import_bundle/robot_setting.txt)** — 那份就是要貼進 EAP「機器人設定」的全文。本檔不再整段複製，避免兩份走鐘（單一真實來源規則）。

設計重點（為什麼這樣寫，對應簡報 P10 情境工程）：

- **角色／語氣** — 對散戶說話、白話翻譯術語、不給買賣建議、先結論再展開 3 點
- **知識邊界** — Graph（16 檔 ETF 的結構化資料）vs Vector（公開說明書等 PDF），講清楚各自管什麼
- **答題決策樹** — 時間／排序／數字 → Graph；為什麼／定義／差異 → Vector；兩者都有 → Hybrid
- **數字鐵則** — 百分比／金額一律走 Graph、不准 LLM 心算；查無資料就答「尚無此資料」，不估算
- **Cypher 約束** — Neo4j v5、禁用 `OVER()`、排名改 `WITH ... COLLECT ... ORDER BY`
- **引用回鏈** — 每答末尾固定附 Graph／Vector／法規三段來源
- **拒答條款** — 選股、進出場時機、預測漲跌 → 直接回拒

### 類別 A：精準檢索（純 Graph）

**範例題**：「台股 ETF 中對台積電持股比重前 5 高的有哪些？」

**路徑**：Graph 直查 → 不需 Vector。

**Cypher**：
```cypher
MATCH (e:ETF)-[h:HOLDS {date: date('2026-06-05')}]->(s:Stock {ticker:'2330'})
RETURN e.ticker, e.name, e.issuer_name,
       round(h.weight_pct * 100) / 100.0 AS tsmc_weight_pct
ORDER BY tsmc_weight_pct DESC
LIMIT 5;
```

**預期回答骨架**：
```
結論：對台積電持股最重的是被動市值/科技型 ETF，最高的 0052 富邦科技達 64.12%。

排行（真實持股 2026-06-05）：
1. 0052 富邦科技 — 台積電 64.12%
2. 006208 富邦台50 — 57.94%
3. 0050 元大台灣50 — 57.91%
4. 00922 國泰台灣領袖50 — 40.00%
5. 00881 國泰台灣5G+ — 38.04%

洞察：你以為買「科技 ETF（0052）」是分散押注科技股，其實近 2/3 是台積電一檔。

來源：
- Graph：(ETF)-[HOLDS]->(Stock{ticker:'2330'})，as of 2026-06-05（真實成分股，holdings_2026-06-05.csv）
```

### 類別 B：深度科普（純 Vector）

**範例題**：「『收益平準金』是什麼？為什麼有人說高股息 ETF 的配息有陷阱？」

**路徑**：Vector 檢索 → 從公開說明書「配息政策」章節 + 主管機關說明拉 chunks → 白話化合成。

**Vector retrieval spec**：
```python
filter = {"doc_type": ["prospectus", "press_release"],
          "section":  ["配息政策", "收益分配", "風險揭露"]}
top_k = 6
rerank = True   # 用 Cohere reranker 提精度
```

**預期回答骨架**：
```
一句話：收益平準金不是「公司賺的錢」，而是新進投資人申購時把錢放進「平準帳戶」，再分配回給所有受益人 — 本質上是把你自己的錢還給你。

3 點細節：
1. 機制：當基金規模快速擴張，新進資金會稀釋每單位淨值；為避免這稀釋效應「吃掉」既有受益人的配息來源，主管機關允許從新進申購的資金中提撥一部分作為「平準金」分配
2. 為什麼像陷阱：投資人看到「年化配息率 10%」興奮買進，但若該配息有大比例來自平準金，等於「左手換右手」，淨值會同步下降，總報酬未必有看到的那麼高
3. 怎麼判讀：每次配息公告會標明「股利所得 / 資本利得 / 收益平準金」三類比例；長期收益平準金 > 30% 時要特別留意

範例（Vector 檢索回的真實公告原文）：
> 國泰永續高股息 00878 收益分配公告（112/10/31）§預估收益分配組成占比：股利所得 31.36%、利息 0.07%、收益平準金 5.71%、已實現資本利得 62.86%。原文：「本基金之配息來源可能為收益平準金……」

延伸：想看你手上 ETF 的配息來源拆解嗎？我可以幫你比較 00878、00919、00929 三檔配息公告的收益平準金占比。

來源：
- Vector：國泰永續高股息 00878 收益分配公告（112/10/31，vector_pdfs/00878_dividend_112Q4.pdf）
- Vector：證期局 2023/11/1 ETF 配息申報新制 https://www.twse.com.tw/market_insights/zh/detail/ff8080818bf08529018bf6c5d9550014

風險提示：過去配息來源不代表未來。
```

### 類別 C：跨文件關聯（Hybrid）⭐⭐⭐ — Demo 主秀

**範例題**：「我有 0050、006208、00878、00992A 各 10 萬，台積電真實曝險多少？為什麼會這麼集中？」

**路徑**：
1. **Graph**：Q2 Cypher 算跨多檔合併曝險（見 `graph_schema.cypher`）
2. **Vector**：對曝險前 1 名個股（台積電）所在的 4 檔 ETF，各抽公開說明書「投資策略 / 篩選條件」段落
3. **LLM 合成**：解釋為何集中（指數共用、ESG 評選台積電優先、主動式重押）

**預期回答骨架**：
```
結論：你這 4 檔 ETF 給了你 31.1% 的台積電曝險（總投入 40 萬，台積電佔 12.42 萬）。集中原因有 3 個。

穿透流向圖：
{Mermaid graph LR 節點：左 3 檔有台積電曝險的 ETF（00878 對台積電 0%，以表格揭露）→ 中 台積電節點}

3 個集中原因：
1. **0050 直接 57.91%**（市值加權，追 FTSE TWSE Taiwan 50；台積電市值占權重最大）
2. **006208 同樣追 FTSE TWSE Taiwan 50，台積電權重 57.94%**（兩檔追同一指數，互相不分散）
3. **00992A 主動式目前台積電 8.38%；而 00878 國泰永續高股息實際不持台積電（0%）**，加它無稀釋作用

延伸動作：要看（a）加買 00940 是分散還是雪上加霜？（b）這 4 檔配息政策差異？

來源：
- Graph：4 ETF × HOLDS × Stock，2026-06-05 真實持股
- 指數事實：0050 / 006208 同追 FTSE TWSE Taiwan 50（公開資訊）

風險提示：過去績效不代表未來；持股資料 as-of 2026-06-05；本回答不構成投資建議。
```

## 3. 子任務 Prompts（資料管線中的 LLM 抽取，必須附 prompt — 鐵則 5.1）

### 3.1 個股「真實主題」標註（漂綠檢測核心，寫回 `(Stock)-[BELONGS_TO_THEME]->(Theme)`）

```
你是台股產業分析師。輸入：個股 ticker、公司名、最新年報「主要產品/業務」段落、近 4 季營收結構百分比。
輸出 JSON：
{
  "ticker": "...",
  "themes": [
    {"name": "電動車", "weight": 0.0~1.0, "confidence": 0.0~1.0,
     "evidence": "原文金句或營收數字"},
    ...
  ]
}
規則：
- weight 是該主題占公司「真實業務」的比例（從營收結構推估），不是市場印象
- 「電動車」需含 EV 整車 / 動力電池 / 馬達 / 充電樁；不含一般車用半導體（除非 > 50% 營收來自 EV 客戶）
- confidence < 0.5 時不輸出該主題
- evidence 必引用原文，不可改寫
主題詞表（受控）：電動車、AI半導體、雲端伺服器、5G、ESG高股息、金融、生技、傳產、綠能、消費、其他
```

### 3.2 ETF 月報觀點抽取（主動式 ETF，寫回 `MonthlyView`）

```
你是 ETF 月報摘要員。輸入：某主動式 ETF 某月月報「投資觀點 / 展望」段落。
輸出 JSON：
{
  "etf": "00992A",
  "month": "2026-04",
  "sentiment": "看多|中性|看空",
  "key_topics": ["AI 雲端需求", "美中關稅", ...],
  "quote": "原文最有代表性的一句"
}
sentiment 判斷：
- 「強勁／顯著成長／優於預期」=看多
- 「審慎樂觀／能見度尚可／持平」=中性
- 「審慎／保守／不確定性高／逆風」=看空
```

### 3.3 配息來源拆解（從配息公告 PDF）

```
你是配息公告解析員。輸入：某 ETF 某次除息公告 PDF 的「收益分配」段落。
輸出 JSON：
{
  "etf": "...", "ex_date": "YYYY-MM-DD",
  "amount_per_unit": float,
  "source_breakdown": {
    "股利所得":  0.0~1.0,
    "資本利得":  0.0~1.0,
    "收益平準金": 0.0~1.0
  }
}
規則：三類加總須 = 1.0；找不到的類別填 0；公告若分多階段（如已實現+未實現）需合併
```

### 3.4 公開說明書章節切塊（給 Vector ingestion）

```
你是 PDF 章節分割器。輸入：ETF 公開說明書全文。
輸出每個 chunk 的 metadata：
{
  "etf": "...",
  "doc_type": "prospectus",
  "section": "投資策略|配息政策|風險揭露|費用結構|追蹤誤差|申購買回|其他",
  "page": int,
  "text": "原文（500 token 以內，章節邊界優先於 token 上限）"
}
章節判定鍵字（受控詞表）：
- 投資策略：「投資目標」「投資政策」「成分股篩選」
- 配息政策：「收益分配」「配息」「除息」「平準金」
- 風險揭露：「風險」「警語」
- 費用結構：「經理費」「保管費」「總費用率」「TER」
- 追蹤誤差：「追蹤誤差」「Tracking Error」「複製方法」
不在受控詞表中的章節 → "其他"
```

## 4. 與 EAP Chat API 的串接點（給 P9 架構圖預埋）

| 動作 | 端點 | Prompt 進場時機 |
|---|---|---|
| 建聊天室 | `POST /api/v1/chat/create` | Robot Setting 在 EAP 後台先設好；建室時帶 project_id |
| 送問題 | `POST /api/v1/chat/{chat_id}` | 使用者問題 → EAP 內部已套 Robot Setting；前端不重複塞 |
| 取訊息 | `GET /api/v1/chat/{chat_id}/messages` | 前端 streaming 解析；遇 Mermaid 區塊用 mermaid.js render |
| 圖表生成 | `POST /api/v1/chat/{chat_id}/{message_id}/chartgen` | 可作進階備援；現場 Demo 優先用前端 Mermaid `graph LR` / 表格，穩定性較高 |
